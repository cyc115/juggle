## Suggested Improvements (1–3)

### 1. Worktree liveness sentinel before GC (`juggle_cmd_integrate.py:374–376`)

The harness GC race is reproducible (3 confirmed agents affected) and produces silent false failures. The fix is a sentinel file: when the test suite starts in a worktree, write a `<worktree>/.juggle-active` lockfile and remove it on exit. `integrate` checks for this file before calling `worktree remove --force`:

```python
# juggle_cmd_integrate.py near line 373
sentinel = Path(worktree_path) / ".juggle-active"
if sentinel.exists():
    return _fail(
        f"Worktree {worktree_path} has an active child process (.juggle-active sentinel). "
        "Wait for it to finish or remove the sentinel manually."
    )
subprocess.run(["git", "-C", main_repo_path, "worktree", "remove", "--force", worktree_path], ...)
```

The sentinel can be written by `juggle_harness.py` (or the `juggle verify` entry point) and is guaranteed to clear when the process exits. No new tables needed.

---

### 2. Mark TODO L13 complete and tighten `MAX_BG_VERIFY_SPAWNS` heuristic (`juggle_verify_cap.py:25`)

The verify-loop code cap landed (cc09651) but is not marked done in TODO.md, and `MAX_BG_VERIFY_SPAWNS = 2` is permissive — an agent can still burn two background cycles (each potentially 100k+ tokens) before being blocked. The sanctioned flow is exactly ONE foreground run; any background spawn is already anomalous:

```python
# juggle_verify_cap.py:25
MAX_BG_VERIFY_SPAWNS = 1  # one background spawn is forgivable; the second is the loop
```

Drop it to 1 and mark the TODO item `[x]`. The DENY message already explains the correct path (`juggle verify` foreground); agents that hit this limit once will adapt.

---

### 3. Re-entrant single-integrate guard at the process level (`juggle_integrate_lock.py`)

The current lock serializes at the DB row level but allows N OS processes to race to acquire it in rapid succession. A process-level advisory lock (e.g., `fcntl.flock` or a PID file with staleness check) would prevent the concurrent-integrate pileup from happening at all, rather than detecting the FF-race after the fact. The fix lives in `juggle_integrate_lock.py` — add a `acquire_exclusive(timeout_secs=120)` wrapper around the existing DB lock that uses `fcntl.flock(LOCK_EX | LOCK_NB)` on the lock file itself and fails fast with a clear "another integrate is running" message if contention is detected.
