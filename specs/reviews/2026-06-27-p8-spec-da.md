# Devil's-Advocate Review — P8 Unified-Topic-Graph Design

_2026-06-27 · reviewer: DA agent (thread EH) · subject: `specs/2026-06-18-unified-topic-graph.md` + as-built P8 code_

**User's explicit bar:** the design must be **clean and maintainable**. This review hunts only for what makes it not.

---

## TL;DR — the one sentence

The spec promised "collapse three tables into ONE, a single state machine, a single source of truth." The **as-built reality is the exact inverse**: the system now maintains **both** models at once (dual-write on every create/update, dual-read in cockpit + orphan_guard), the **legacy `db_graph` engine still drives all task execution**, the new unified `nodes` table is a passive **follower-mirror** that is manually re-synced from the legacy computation, and the centerpiece unified state machine (`db_node_machine.node_transition`) is **dead code** (imported once with `# noqa: F401`, never called). P8 is currently sitting at the **maximum-debt midpoint** of a strangler migration — 2× the surface area, none of the simplification — with **no test-green incremental path to finish** (proven by the wave-3 floor analysis: the flip is blocked on 4 unresolved decisions).

13 findings: **3 CRITICAL, 5 HIGH, 4 MED, 1 LOW**. The three CRITICALs are not independent bugs — they are three faces of the same root cause: *a second data model was added before the first was retired, with no forcing-function to ever retire it.*

## C1 — CRITICAL — The unified state machine is dead code; the legacy `db_graph` engine still runs everything

**The smell:** leaky/aspirational abstraction. The spec's centerpiece (§4.5: a single `node_transition(db, node_id, event)` that is "the SOLE writer of `node.state`") **does not drive a single production code path.** It was built (P2), tested, and then orphaned.

**Evidence:**
- `src/dbops/db_node_machine.py` defines the unified `node_transition(state, event, kind)`. Its only importer is `src/juggle_add_node.py:18`:
  `from dbops.db_node_machine import node_transition, _KIND_LEGAL  # noqa: F401` — the `# noqa: F401` literally documents that it is **imported but unused**.
- The real task engine is still `src/dbops/db_graph.py:49 _TRANSITIONS` (the `pending`-based legacy machine). Every executor path calls it: `juggle_graph_dispatch.py:298`, `juggle_cmd_agents_graph.py:215`, `juggle_graph_add.py:213`, `juggle_graph_load.py:162`, and `juggle_add_node.py:263` itself.
- `juggle_add_node.py:258-274` is the tell: after inserting the node it calls `db_graph.recompute_ready()` (legacy), reads `db_graph.get_task()` (legacy `graph_tasks`), and **mirrors that result back** into `nodes.state` via `_update_node_state`. The unified machine never decides anything.
- The spec's API surface (`db.get_node`, `db.update_node`, a `dbops/db_nodes.py`) **does not exist** — `grep -rn "def get_node\b" src/` returns nothing.

**Why it hurts most:** the entire value proposition ("one transition function, one `state` column") is unrealized, yet the dead machine sits next to the live one inviting every future reader to use the wrong one. You cannot drop `graph_tasks` while `db_graph` is the engine, so this directly blocks completion.

**Fix direction:** make `node_transition` authoritative OR delete it until it is. Cleanest end-state: `db_graph.transition()` becomes a thin shim that calls `db_node_machine.node_transition` and writes `nodes` (not `graph_tasks`); `add_node` computes readiness via the unified machine, not `db_graph.recompute_ready`. Until then, an unwired "single source of truth" is worse than no abstraction.

**Agent-verify:**
- After fix, `grep -rn "node_transition(" src/ | grep -v "def node_transition" | grep -v "noqa"` must return **> 0** real call sites.
- `grep -n "db_graph.recompute_ready\|db_graph.get_task" src/juggle_add_node.py` must return **empty**.
- A test drives a task node `open→ready→…→done` through `node_transition` only and asserts `graph_tasks` is never written.

## C2 — CRITICAL — Dual-write + dual-read of two full data models; the "collapse" widened the system

**The smell:** dual-write debt with no completion forcing-function. A strangler-fig migration parked at its most expensive point — both vines alive, gripping every read and write.

**Evidence (writes):** `juggle_add_node._add_task_node` writes **four** tables for one logical node — `nodes` (`:208`), `graph_tasks` (`:217`), `node_edges` (`:225-234`), **and** `graph_edges` (`:236-246`) — with **two** edge stores and **two** state vocabularies (`nodes.state='open'` at `:212` vs `graph_tasks.state='pending'` at `:220`). Conversation writes fan out too: `dbops/threads.py` calls `conv_node_mirror.mirror_conv_update` at **5 sites** (`:274,304,316,344,381`) plus `mirror_conv_insert` (`:170`), each a best-effort write that **silently swallows** a missing-table error (`conv_node_mirror.py:32` `except sqlite3.OperationalError: pass`).

**Evidence (reads — RK5 made permanent):** the system reads *both* models in the same modules:
- `juggle_cockpit_model.py:218` reads `FROM nodes` for the DAG, while `:264-290` read `FROM threads` for the conversation panels.
- `dbops/orphan_guard.py:89` reads `FROM nodes`, then `:52` and `:125` fall back to `SELECT thread_id FROM graph_topics` ("compat: dual-write still active").

**Why it hurts:** every create/update now costs two writes that must agree, kept in sync by hand and by silent `except: pass`. The mental model the spec wanted to *delete* (3-tier) is now a **6-surface** model (threads + graph_topics + graph_tasks + nodes + node_edges + graph_edges) that a maintainer must hold simultaneously. This is the canonical "we added the new thing but never removed the old thing" failure.

**Fix direction:** one authoritative writer per kind. Either (a) finish the flip cluster-atomically so legacy writes are deleted, or (b) until then, make the *non-authoritative* side a SQLite `VIEW` over the authoritative table (no hand-synced second write at all). Recommended: (a) with the cluster boundaries from H3. Remove every `except sqlite3.OperationalError: pass` that hides a divergence.

**Agent-verify:**
- Post-collapse: `grep -rn "INSERT INTO graph_tasks\|INSERT INTO graph_edges\|INSERT OR IGNORE INTO graph_" src/juggle_add_node.py` → **empty**.
- `grep -rn "FROM threads\|FROM graph_topics\|FROM graph_tasks" src/juggle_cockpit_model.py src/dbops/orphan_guard.py` → **empty**.
- A test asserts a single `add-node --kind task` produces exactly **one** row in exactly **one** of {`nodes`, `graph_tasks`} (not both).

## C3 — CRITICAL — Two state vocabularies inside the *same* `nodes` table, and the unified machine cannot process one of them

**The smell:** vocabulary inconsistency baked into the data, directly contradicting goal §1 ("one `state` column — no per-table status enum divergence"). The divergence didn't get eliminated; it got *moved inside* the unified table.

**Evidence:**
- `migration_nodes_parity.backfill_graph_parity` (`:98-101, :110-115`) actively **rewrites `nodes.state` `'open'→'pending'`** for `kind='task'` rows whose legacy row was `pending`, "because `db_graph` queries `state='pending'`."
- So `kind='conversation'` nodes carry `'open'` (`conv_node_mirror.py:29`) while `kind='task'` nodes carry `'pending'` — **two entry vocabularies in one column.**
- The unified machine has **no `('pending', …)` transitions at all** (`db_node_machine.py:19-60` starts at `('open', 'deps_ready')`). Therefore `node_transition('pending', 'deps_ready', 'task')` **raises `InvalidTransition`.** The unified machine is structurally incompatible with the data the migration produces — another nail in C1.

**Why it hurts:** the single most-quoted goal of the design is falsified *inside the unified artifact itself*. Any future code that trusts "nodes.state speaks the new vocab" breaks on task rows.

**Fix direction:** pick ONE task-entry state and make it global. Recommended: migrate `db_graph` and all task-state queries (`db_graph.py:136,256,279`) from `'pending'` to `'open'`, delete `'pending'` everywhere, and delete `backfill_graph_parity`'s `open→pending` correction. The wave-3 "keep pending" choice (Q3=A) bought short-term test-green at the cost of a *permanent* vocab split — for a clean end-state that trade is wrong; do the rename once.

**Agent-verify:**
- `grep -rn "'pending'" src/` → empty, except inside a single dated, never-run historical migration.
- `SELECT DISTINCT state, kind FROM nodes` never returns `pending` for any kind.
- A test: insert a task node, then drive it via `node_transition` from its actual stored `state` — must not raise.

## H1 — HIGH — The "single source of truth" value-map is triplicated, plus a divergent fourth copy

**The smell:** copy-paste of the one thing that was explicitly supposed to exist once. `node_translation.py`'s own docstring says "the value map is defined exactly once" — **that claim is false.**

**Evidence — the identical `thread.status → state` dict appears 3×:**
- `dbops/node_translation.py:5 STATUS_TO_STATE`
- `dbops/db_node_machine.py:111 _THREAD_STATUS_TO_NODE_STATE` (byte-identical mapping, different name)
- `dbops/migrations_nodes.py:22 _THREAD_STATUS_MAP` (again identical)
- **+ a 4th, divergent vocab:** `dbops/db_mirror.py:27 _THREAD_TO_STATE = {active:running, idle:pending, done:verified}` — same conceptual input, *different* target states.

**And the reverse map is duplicated across languages:** `node_translation.py:20 STATE_TO_STATUS` (Python dict) vs `:33 STATE_AS_STATUS_SQL` (`CASE state WHEN 'open' THEN 'active' WHEN 'done' THEN 'closed' …`). Two hand-synced encodings of the same reverse mapping; nothing tests that they agree.

**Why it hurts:** the vocab map is the load-bearing seam of the whole migration. Four copies means a fifth status (or a tweak) must be edited in four places or it silently diverges — exactly the failure `db_mirror`'s drift already demonstrates.

**Fix direction:** one module owns both directions; `db_node_machine`, `migrations_nodes`, and `db_mirror` import from it; delete the local copies. Generate `STATE_AS_STATUS_SQL` from `STATE_TO_STATUS` (or add a test asserting key-by-key equivalence).

**Agent-verify:**
- `grep -rn '"active": "open"' src/` → exactly **1** hit.
- A test asserts `{k: STATE_TO_STATUS[k] for k}` matches the SQL `CASE` output for every state via an in-memory SQLite eval.

## H2 — HIGH — The alias-shim is built-but-unused; if adopted it makes `nodes` lie in the legacy vocabulary forever

**The smell:** alias-shim as a permanent crutch, presented as a "clean boundary." Right now it is *speculative dead infrastructure*; the moment it's wired it becomes *hidden permanent debt*.

**Evidence:**
- `node_translation.py:32-37` defines `STATE_AS_STATUS_SQL` / `CONV_ALIAS_SHIM` (`SELECT *, CASE … AS status, title AS topic, last_active_at AS last_active`). **No read-path module imports them** — `grep -rln "CONV_ALIAS_SHIM\|STATE_AS_STATUS_SQL" src/` returns only `node_translation.py` itself.
- Its stated purpose (wave-3 Q1=A) is to let "~107 consumers that read `row['status']`/`'topic'`/`'last_active'` keep working **untouched**." That means the read-collapse is explicitly designed to **not** clean up the consumers — it teaches the unified table to re-emit the *old* column names and *old* status values forever.

**Why it hurts:** a shim that re-aliases `state→status`, `title→topic`, `last_active_at→last_active` on every read is not a boundary — it's a translation layer that guarantees the codebase never speaks one vocabulary. "Clean" requires the consumers to actually adopt the new names; the shim's whole reason for existing is to avoid that.

**Fix direction:** decide explicitly. Recommended (a): do the one-time mechanical rename of the ~107 consumers to `state`/`title`/`last_active_at`, delete the shim — greppable to zero, done. If (b) the shim stays, it must be the **sole** read path with a ratchet test forbidding any *new* `row['status']` access. Half-wired-and-unused (today's state) is the worst of both.

**Agent-verify:**
- If (a): `grep -rn "\['status'\]\|\['topic'\]\|\['last_active'\]" src/` → empty; `CONV_ALIAS_SHIM` deleted.
- If (b): every `SELECT * FROM nodes WHERE kind='conversation'` is suffixed with the shim (lint/test), and a ratchet test pins the consumer count at its current number, never higher.

## H3 — HIGH — No test-green incremental flip exists; the §13 phasing does not match the real dependency graph

**The smell:** the plan oversells "strangler-fig, tests green at every boundary" (§1, §13) when the ground truth is the opposite — the flip is one big gated step.

**Evidence:** `research/2026-06-22-p8-wave3-collapse-floor.md` proves (with an atomicity argument) that the minimum test-green unit is a **whole cluster**, and every cluster is blocked on a USER decision:
- Q1 — `SELECT *` key-rename fans out to **~107 consumer sites** the spec's repoint table never scoped.
- Q2 — `graph_*.thread_id` has **no `nodes` home** (load-bearing; `parent_id` ≠ that relation).
- Q3 — task-state `pending`/`open` vocab drift (see C3).
- Q4 — dead `juggle_migrate_lifecycle.py` (9 refs) is a permanent floor (see L1).
- Conversation **status writes are not dual-written** at the source (`threads.py` historically wrote only `threads.status`), so any conversation read flipped to `nodes` reads stale state unless the write flips in the *same* commit → the whole conversation cluster is one atomic unit.

The spec's per-phase RED-test lists (P1–P8) assume independently-shippable slices; the real graph has two monolithic clusters gated on decisions, so several of those phase boundaries are not actually green-able as written.

**Why it hurts:** an implementer following §13 hits a wall at the first real flip and discovers the phasing is fictional — wasted work and a likely un-reviewable mega-commit (which is exactly what cyc_EC refused to ship).

**Fix direction:** rewrite §13 around the *real* atomic clusters (conversation cluster; graph cluster) with Q1–Q4 resolved up front and recorded in the spec. Mark which P-phase RED tests are now obsolete. Treat "tests green at every boundary" as a per-cluster promise, not per-phase.

**Agent-verify:** `juggle doctor --pre-p8-check --json` reports `static.fail` decreasing **monotonically** across cluster commits; each cluster commit lands with full `pytest` green (CI gate per commit, not per phase).

## H4 — HIGH — The authoritative DDL lies; `CREATE_NODES` omits 4 live columns and the mirror silently drops data

**The smell:** schema drift between the "source of truth" DDL and the real table; a silent `except` that converts a schema gap into invisible data loss.

**Evidence:**
- `dbops/schema_nodes.py` docstring: *"Owns: CREATE TABLE strings for nodes."* But `CREATE_NODES` (`:8-57`) has **no** `user_label`, `assigned_by`, `last_active_at`, or `dispatch_thread_id`. Those exist only via `migration_nodes_parity.py:12-18` ALTERs (Migration 50).
- `conv_node_mirror.mirror_conv_insert` (`:24-31`) **INSERTs `user_label` and `last_active_at`** into `nodes`; the unique slug index (`migration_nodes_parity.py:36-38`) needs `user_label`. On any DB where Migration 50 hasn't run, that INSERT raises `OperationalError` — which `:32` **swallows silently** → the conversation node is **never created**, with no error surfaced.
- To know the real `nodes` columns you must read `schema_nodes.py` **and** `migration_nodes_parity.py`. The canonical DDL is incomplete by design.

**Why it hurts:** the one file a new maintainer reads to learn the schema is wrong by 4 columns, and the gap degrades to silent data loss instead of a loud failure. Both are textbook un-maintainable.

**Fix direction:** fold the 4 parity columns into `CREATE_NODES` so a fresh DB built from the DDL alone is complete (keep the additive ALTERs for upgrades — they're idempotent no-ops once the DDL has them). Narrow or remove the blanket `except OperationalError: pass` in `mirror_conv_insert` so a real schema gap fails loud.

**Agent-verify:** a test creates a fresh table from `CREATE_NODES` **only** (no migrations) and asserts every column `conv_node_mirror` writes (`user_label`, `last_active_at`, `session_id`, …) is present; `PRAGMA table_info(nodes)` after `CREATE_NODES` ⊇ the mirror's column set.

## H5 — HIGH — The LOCKED spec is now stale vs the build; it actively misleads future agents

**The smell:** documentation rot on a document marked authoritative. §15 "Spec Self-Review" asserts "Placeholders: None / consistency check ✓ / no ambiguity," but the implementation diverged from the spec on **every hard call** and the spec was never updated.

**Evidence of divergence (spec says X; code does Y):**
- §3.1 + §10 say `graph_topics.thread_id` is **eliminated** ("node IS the thread"). Code **added `nodes.dispatch_thread_id`** to carry it (`migration_nodes_parity.py:17`). (See M1.)
- §4.1/§4.5 say `state` uses `open` and one machine writes it. Code keeps `pending` for tasks and the machine is unused. (See C1/C3.)
- §8.1 says "Forward-only. No dual-read compatibility layer." Code has dual-read in cockpit + orphan_guard. (See C2.)
- §6/§9 describe `add-node` as a clean creation verb; the as-built path is a 4-table dual-write that defers to the legacy engine. (See C1/C2.)

**Why it hurts:** a `LOCKED` spec is the first thing the next implementer trusts. Here it will lead them to write `node_transition`-based code that breaks on `pending`, or to assume `thread_id` is gone. A wrong authoritative doc is worse than no doc.

**Fix direction:** demote the 2026-06-18 spec to "superseded" and write a short as-built addendum that states the four resolved decisions (Q1–Q4), the follower-mirror reality, and the remaining flip plan (from H3). Keep one document that matches the code.

**Agent-verify:** spec header no longer says `LOCKED` without an "as-built status" section; a doc test (or reviewer checklist) asserts each §10 "eliminated" item is actually absent from `src/` (it isn't today — `dispatch_thread_id` exists).

## M1 — MED — `dispatch_thread_id` re-introduces the exact coupling the spec said it deleted

**The smell:** an "eliminated" relation resurrected under a new name; the headline simplification ("node IS the thread") is false for task nodes.

**Evidence:** §10 deletion list: *"`thread_id` in graph_topics → Node IS the thread."* Reality: `migration_nodes_parity.py:16-17` adds `nodes.dispatch_thread_id` and backfills it from `graph_tasks.thread_id` / `graph_topics.thread_id` (`:90-109`) because the task→agent-thread link "has no other nodes home" (wave-3 Q2). `parent_id` is the task→topic relation, a *different* edge. So a task node still points at a separate dispatch-thread row.

**Why it hurts:** the conceptual win the design was sold on doesn't hold; worse, it's a nullable foreign-key-ish column with no FK constraint and no model documentation, so its meaning lives only in a migration comment.

**Fix direction:** model the dispatch relation explicitly — either a typed edge in `node_edges` (add an edge-kind), or carry it on the agent row (`agents.assigned_node`) where the binding actually lives. If the column stays, give it a real `REFERENCES nodes(id)` and document it in the schema, and update §10 to stop claiming elimination. Post-collapse cleanup.

**Agent-verify:** schema documents `dispatch_thread_id` (or its replacement edge) with a comment + constraint; a test asserts the task→dispatch-thread link round-trips through the chosen model.

## M2 — MED — One wide sparse table, ~15 kind-specific NULL columns, zero discriminator enforcement

**The smell:** kind-discriminator hygiene. A single wide table is a *defensible* choice for this orchestrator (avoids joins, SQLite-friendly) — the smell is that it's **unenforced and undocumented**, so nothing stops a malformed cross-kind row.

**Evidence:** `schema_nodes.CREATE_NODES` mixes conversation-only columns (`session_id, summary, key_decisions, open_questions, last_user_intent, summarized_msg_count, show_in_list`, + `user_label` from M50) with task-only columns (`verify_cmd, worktree_path/branch, main_repo_path, merged_sha, handoff, diffstat, verified_at`). ~15 columns are NULL for the wrong kind. There is **no `CHECK` constraint and no guard** preventing a `kind='conversation'` row with `verify_cmd` set, or a `kind='task'` row with `key_decisions`. Discriminator correctness is by-convention only.

**Why it hurts:** wide sparse tables are fine *when the discriminator is enforced*. Here a buggy writer can silently produce a nonsense row (e.g. a conversation with a `verify_cmd` the tick might one day read), and nothing catches it. Verdict: **keep the single table** (splitting into per-kind tables would re-introduce the joins the design correctly killed) — but make the discriminator real.

**Fix direction:** add `CHECK` constraints (`kind='conversation' OR verify_cmd IS NULL`, etc.) or a single insert/update guard, and document the wide-table decision in the spec as deliberate. Do **not** split the table.

**Agent-verify:** a test inserts each illegal cross-kind combination and asserts it's rejected; `SELECT COUNT(*) FROM nodes WHERE kind='conversation' AND verify_cmd IS NOT NULL` is pinned at 0.

## M3 — MED — Conversation/task state-write atomicity: `nodes.state` drifts from the legacy authority

**The smell:** non-atomic cross-model sync — a window where the mirror disagrees with the authority.

**Evidence:** `juggle_add_node._add_task_node` commits the `nodes` + `graph_tasks` insert in one transaction (`:248`), then **after commit** runs `db_graph.recompute_ready()` (`:263`) and `_update_node_state()` (`:270`) on **separate connections**. A crash or error between `:248` and `:270` leaves `nodes.state='open'` while `graph_tasks.state='ready'`. Readers of `nodes` (cockpit DAG `cockpit_model.py:218`) then show a stale state. (Conversation writes via `conv_node_mirror` *are* in-transaction — good — but the task path is not.)

**Why it hurts:** the mirror's correctness depends on two post-commit steps succeeding; the design has no reconciliation for the gap, so drift is silent and sticky until the next write.

**Fix direction:** compute readiness and write `nodes.state` **inside the same transaction** as the insert; or, until the flip, make `nodes.state` for tasks a *derived read* from `graph_tasks` (a view) rather than a separately-written copy. Add a regression pin for the crash-between window.

**Agent-verify:** a test inserts a no-dep task and asserts `nodes.state == graph_tasks.state` in a single read immediately after `add_node` returns, with no intervening tick.

## M4 — MED — Gate-A static scanner has blind spots; it can report `PASS:0` while live coupling persists

**The smell:** a safety gate that can give false confidence — the worst kind, because the drop is irreversible (§12.2 after P8).

**Evidence:** `p8_readiness.scan_legacy_refs` (`:93-106`) + `_excluded` (`:77-83`) **skip every `dbops/schema*` and `dbops/migration*` file and every comment-only line**, and `_GATE` (`:71-74`) only matches literal `FROM|JOIN|INTO|UPDATE|DELETE|REFERENCES <table>`. So legacy coupling that lives in an excluded module (`db_mirror.py` writes `graph_topics`; `db_topics.py`), or is reached *through* a helper that names the table in an excluded file, or is built by string-interpolation, is **invisible to the gate**. Gate B checks row-mirroring but not code-reachability. A `PASS:0 / READY` can therefore coexist with `db_graph`/`db_mirror` still being imported and run.

**Why it hurts:** the gate is the thing standing between "drop tables" and "lose data." Its exclusions are pragmatic but unstated, and they overlap exactly with the modules that still drive the legacy engine (C1).

**Fix direction:** strengthen Gate A to also assert **no module imports `db_graph` / `db_mirror` / `db_topics`** (the legacy engines) after collapse — import-graph reachability, not just SQL-literal grep. Log every excluded file in the report so a reviewer sees what the gate did *not* check.

**Agent-verify:** `juggle doctor --pre-p8-check --json` includes an `excluded_files` list and an `import_refs` count for the legacy engine modules; both must be 0 (and the list reviewed) before a drop is allowed.

## Cross-cutting recommendation (the clean-design direction)

The findings collapse to one corrective sequence. Do them in this order; each is independently shippable and monotonically reduces surface:

1. **Pick one task-state vocab (`open`) and one engine.** Rename `db_graph`'s `pending→open` (C3), then make `db_graph.transition` delegate to `db_node_machine.node_transition` and write `nodes` (C1). Delete `backfill_graph_parity`'s vocab correction.
2. **Centralize the vocab maps** in `node_translation` and delete the 3 copies; generate the SQL `CASE` from the dict (H1).
3. **Resolve Q1 by renaming the ~107 consumers** to `state/title/last_active_at` and **delete the alias-shim** (H2) — this is the clean end-state, not a shim.
4. **Flip cluster-atomically** (conversation, then graph) per the real dependency graph; delete the legacy *writes* and *reads* as each cluster lands (C2). Update §13.
5. **Make the DDL honest** (H4) and **the gate honest** (M4) before the irreversible drop.
6. **Then** the post-collapse cleanups: `dispatch_thread_id` modeling (M1), `CHECK` constraints (M2), atomicity pin (M3), delete `migrate_lifecycle` (L1), rewrite the spec as as-built (H5).

The single deepest smell to take away: **a second data model and a second state machine were introduced before the first were retired, and nothing in the design forces retirement.** Every CRITICAL is a symptom of that. "Clean and maintainable" is unreachable until exactly one model and one machine remain — the migration must be *finished*, not just *started safely*.
